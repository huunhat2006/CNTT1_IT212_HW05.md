1. Nguyên Lý Hoạt Động Của Các Phương Án
Phương án 1: Đối soát trong bộ nhớ (In-Memory Processing)
Nguyên lý: Hệ thống sử dụng một ứng dụng Java (Worker/Service) độc lập. Hằng ngày, ứng dụng này sẽ kích hoạt một tiến trình Batch (ví dụ: Spring Batch) để kéo toàn bộ dữ liệu từ 2 nguồn (Core Banking và Visa via file CSV/ISO 8583) về.

Dữ liệu từ Core Banking được nạp vào một HashMap với Key là mã đối chiếu (như Retrieval Reference Number - RRN hoặc Transaction ID) và Value là đối tượng giao dịch.

Sau đó, luồng dữ liệu từ Visa sẽ được đọc tuần tự (Stream-read). Với mỗi giao dịch Visa, hệ thống sẽ thực hiện hàm .get(transactionId) trên HashMap để đối chiếu trạng thái, số tiền. Kết quả (Khớp, Lệch Core, Lệch Visa) sẽ được ghi xuống file kết quả hoặc DB sau cùng.

Phương án 2: Đối soát bằng SQL JOIN tại Database
Nguyên lý: Hệ thống thiết lập 2 bảng tạm (Staging Tables) trong Cơ sở dữ liệu: staging_core_transactions và staging_visa_transactions. Dữ liệu thô từ hai nguồn sau khi nhận được sẽ được Import thẳng vào hai bảng này (sử dụng các công cụ Bulk Insert tốc độ cao).

Sau khi nạp xong, hệ thống sẽ chạy các câu lệnh SQL Heavy-Query sử dụng phép toán FULL OUTER JOIN, LEFT JOIN kết hợp các điều kiện toán học (WHERE, GROUP BY) trực tiếp trên DB Engine để tìm ra các bản ghi chênh lệch.

Kết quả đối soát cuối cùng sẽ được INSERT trực tiếp sang bảng reconciliation_results.

Không có phương án nào tốt nhất, chỉ có phương án phù hợp nhất với hạ tầng hiện tại của SecureBank:

Nên chọn Phương án 1 (In-Memory Processing) khi:
Yêu cầu về thời gian (SLA) nghiêm ngặt: Hệ thống cần hoàn thành đối soát dữ liệu ngày hôm trước trước 6h sáng để mở cổng giao dịch mới, đòi hỏi tốc độ xử lý tính bằng phút.

Hạ tầng Cloud/Microservices: Bạn có khả năng dễ dàng cấp phát các Pod/Container có cấu hình RAM lớn (ví dụ: Kubernetes Pod với 16GB RAM) chạy dạng Ephemeral (chỉ bật lên lúc 2h sáng để đối soát, xong tự tắt để tiết kiệm chi phí).

Có kế hoạch mở rộng: Khi lượng giao dịch tăng lên 20 - 50 triệu giao dịch/ngày trong tương lai, bạn dễ dàng chuyển hướng sang các framework tính toán phân tán (như Apache Spark) dựa trên cùng tư duy in-memory này.

Nên chọn Phương án 2 (SQL JOIN) khi:
Dữ liệu đã nằm sẵn trong Hệ sinh thái Data Warehouse: SecureBank có sẵn các DB chuyên dụng cho phân tích (như PostgreSQL được tối ưu, Oracle Exadata, hoặc BigQuery/Snowflake) có năng lực xử lý tính toán song song ở tầng đĩa cứng (Parallel Execution).

Đội ngũ vận hành mạnh về SQL: Hệ thống cần thay đổi luật đối soát (Business Rules) liên tục. Việc sửa một câu lệnh SQL JOIN hoặc cấu hình bảng metadata rule rõ ràng là nhanh và ít rủi ro deploy lại code hơn là sửa mã nguồn Java.

Ngân sách hạ tầng ứng dụng hạn chế: Không muốn đầu tư hoặc duy trì các cụm Server Java cấu hình mạnh, chấp nhận tận dụng khoảng thời gian thấp điểm ban đêm (Off-peak hours) của Database để xử lý Batch.

Đề xuất quy trình chuẩn hóa thời gian trước khi đối soát. 

Chào bạn, việc lệch múi giờ (Timezone Mismatch) giữa các nguồn dữ liệu là một trong những "cơn ác mộng" kinh điển nhất trong hệ thống đối soát tài chính toàn cầu. Khi Core Banking chạy múi giờ địa phương (GMT+7) còn các tổ chức thẻ quốc tế như Visa/Mastercard chạy múi giờ chuẩn quốc tế (GMT+0 / UTC), việc đối chiếu thô sẽ dẫn đến những hậu quả nghiêm trọng cho hệ thống.

1. Hậu Quả Khi Đối Soát Trực Tiếp Không Chuẩn Hóa
Nếu hệ thống thực hiện so sánh chuỗi ngày giờ hoặc giá trị timestamp thô mà không đưa về cùng một hệ quy chiếu, các lỗi nghiêm trọng sau sẽ xảy ra:

Lệch ngày đối soát (Date Boundary Overlap): Một giao dịch Visa phát sinh lúc 21:00 ngày 28/06/2026 (GMT+0) thực chất là 04:00 ngày 29/06/2026 tại Việt Nam (GMT+7). Nếu đối soát theo ngày, giao dịch này sẽ bị xếp vào ngày 28 ở file Visa nhưng lại nằm ở ngày 29 ở file Core. Kết quả: Cả hai ngày đều bị lệch đối soát (Unreconciled).

Tỷ lệ báo lỗi giả (False Positives) tăng vọt: Hệ thống đối soát tự động thường có một khoảng dung sai thời gian (ví dụ: hai giao dịch khớp nhau nếu thời gian lệch không quá 5 phút). Khi lệch tới 7 tiếng (420 phút), tỷ lệ khớp tự động bằng 0%, buộc đội ngũ Vận hành (Operations) phải đối soát thủ công hàng triệu giao dịch.

2. Vì Sao Nên Chuẩn Hóa Về UTC?
Trong kiến trúc phần mềm hiện đại, UTC (Coordinated Universal Time) được chọn làm "ngôn ngữ chung" cho thời gian vì:

Tính nhất quán (Consistency): UTC không bị ảnh hưởng bởi giờ mùa hè (DST - Daylight Saving Time) như múi giờ của các quốc gia châu Âu hay Mỹ.

Hỗ trợ đa nguồn (Multi-source integration): Hôm nay SecureBank chỉ đối soát với Visa (GMT+0), nhưng ngày mai có thể tích hợp thêm các cổng thanh toán nội địa (GMT+7) hoặc JCB (GMT+9). Việc đưa tất cả về một trục tọa độ trung gian (UTC) giúp số lượng hàm chuyển đổi luôn là N thay vì N×M.

3. Minh Họa Xử Lý Bằng Java 8 Date Time API
Java 8 (và các phiên bản mới như Java 17/21) cung cấp gói java.time cực kỳ mạnh mẽ để giải quyết triệt để bài toán này thông qua ZonedDateTime và ZoneId.

Dưới đây là đoạn mã minh họa cách đưa hai mốc thời gian lệch múi giờ về cùng một chuẩn UTC để so sánh chính xác:

Java
package com.securebank.util;

import java.time.LocalDateTime;
import java.time.ZoneId;
import java.time.ZonedDateTime;
import java.time.format.DateTimeFormatter;

public class TimeNormalizationDemo {

    public static void main(String[] args) {
        // Định dạng thời gian đầu vào
        DateTimeFormatter formatter = DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");

        // Định nghĩa các ZoneId cần thiết
        ZoneId zoneGmt7 = ZoneId.of("Asia/Ho_Chi_Minh"); // GMT+7
        ZoneId zoneGmt0 = ZoneId.of("UTC");              // GMT+0 (hoặc "Etc/GMT")

        // -----------------------------------------------------------------
        // TH 1: Chuyển đổi thời gian Core Banking (GMT+7) -> UTC
        // -----------------------------------------------------------------
        String coreTimeStr = "2026-06-29 04:00:00"; // 4h sáng ngày 29 tại VN
        LocalDateTime localCoreTime = LocalDateTime.parse(coreTimeStr, formatter);
        
        // Gắn múi giờ gốc GMT+7 cho dữ liệu Core
        ZonedDateTime coreZonedDateTime = ZonedDateTime.of(localCoreTime, zoneGmt7);
        // Chuyển dịch vùng sang UTC
        ZonedDateTime coreInUtc = coreZonedDateTime.withZoneSameInstant(zoneGmt0);

        // -----------------------------------------------------------------
        // TH 2: Chuyển đổi thời gian Visa (GMT+0) -> UTC
        // -----------------------------------------------------------------
        String visaTimeStr = "2026-06-28 21:00:00"; // 9h tối ngày 28 tại tổng doanh Visa
        LocalDateTime localVisaTime = LocalDateTime.parse(visaTimeStr, formatter);
        
        // Gắn múi giờ gốc GMT+0 cho dữ liệu Visa
        ZonedDateTime visaZonedDateTime = ZonedDateTime.of(localVisaTime, zoneGmt0);
        // Chuyển dịch vùng sang UTC (Bản chất vẫn giữ nguyên giờ nhưng chuẩn hóa kiểu dữ liệu)
        ZonedDateTime visaInUtc = visaZonedDateTime.withZoneSameInstant(zoneGmt0);

        // -----------------------------------------------------------------
        // KẾT QUẢ SO SÁNH
        // -----------------------------------------------------------------
        System.out.println("Core Banking (Original): " + coreTimeStr + " (GMT+7)");
        System.out.println("Core Banking (In UTC):   " + coreInUtc.format(formatter));
        System.out.println("---");
        System.out.println("Visa Card    (Original): " + visaTimeStr + " (GMT+0)");
        System.out.println("Visa Card    (In UTC):   " + visaInUtc.format(formatter));
        System.out.println("---");

        // Thực hiện so sánh bằng hàm isEqual của ZonedDateTime
        boolean isSameTime = coreInUtc.isEqual(visaInUtc);
        System.out.println("Hai giao dịch có trùng thời gian không? -> " + (isSameTime ? "CÓ! KHỚP NHAU." : "KHÔNG!"));
    }
}
4. Đề Xuất Quy Trình Chuẩn Hóa Dữ Liệu Khuyến Nghị (Best Practices)
Để đảm bảo hệ thống chạy mượt mà và không phát sinh lỗi tiềm ẩn, SecureBank nên áp dụng quy trình chuẩn hóa thời gian qua 3 bước (Pipeline):

[Nguồn Dữ Liệu Thô] ──> [Tầng ETL / Parser] ──> [Staging DB (Instant)] ──> [Engine Đối Soát]
(Core: GMT+7, Visa: GMT+0)  (Áp dụng Java API)     (Lưu dạng Epoch/UTC)      (So sánh tuyệt đối)
Bước 1: Khai báo Metadata Vùng dữ liệu tại tầng ETL/Parser
Khi đọc file từ bất kỳ nguồn nào, tầng Parser bắt buộc phải gắn kèm thông tin ZoneId của nguồn đó (Core Banking = Asia/Ho_Chi_Minh, Visa = UTC). Tuyệt đối không dùng giờ mặc định của Server (ZoneId.systemDefault()).

Bước 2: Chuyển đổi và lưu trữ dưới dạng thực thể Unix Epoch hoặc UTC
Khi đưa dữ liệu vào các bảng tạm (Staging Tables) ở Database, cột thời gian giao dịch nên được định nghĩa dưới kiểu dữ liệu hỗ trợ múi giờ (Ví dụ: TIMESTAMP WITH TIME ZONE trong PostgreSQL hoặc lưu hẳn thành một số nguyên kiểu BIGINT đại diện cho số miligiây tính từ năm 1970 - Epoch Time). Epoch Time là bất biến toàn cầu, không phụ thuộc vào múi giờ.

Bước 3: Thực thi Engine đối soát trên một trục thời gian phẳng
Khi hai nguồn dữ liệu đã được đưa về cùng một chuẩn Epoch hoặc UTC ở tầng DB, các câu lệnh SQL JOIN hoặc logic Java Collection lúc này chỉ việc so sánh toán học bằng dấu bằng = hoặc so sánh khoảng lệch bằng phép trừ tuyệt đối ABS(core_time - visa_time) <= 300 (dung sai 5 phút) một cách cực kỳ an toàn và chính xác.

1. Algorithm Outline (Phác thảo Thuật toán)
Để đối soát hiệu quả 2 danh sách giao dịch, thuật toán được tối ưu hóa theo các bước sau nhằm đạt độ phức tạp tuyến tính O(N+M) thay vì O(N×M):

Chuẩn hóa & Map hóa nguồn Đối tác (Partner Data Map):

Duyệt qua danh sách partnerTxns.

Chuẩn hóa thời gian của từng giao dịch về chuẩn UTC.

Lưu các giao dịch đã chuẩn hóa vào một HashMap với Key là transactionId để tối ưu hóa tốc độ tra cứu xuống O(1).

Đối soát tuyến tính từ nguồn Core (Core-Driven Reconciliation):

Duyệt qua danh sách coreTxns.

Chuẩn hóa thời gian của giao dịch Core về chuẩn UTC.

Tra cứu giao dịch tương ứng trong Partner Map bằng transactionId.

Kịch bản A (Tìm thấy đối tác): Tiến hành so khớp (Match) các trường dữ liệu: amount, status, và thời gian giao dịch (transactionDate). Nếu có bất kỳ sai lệch nào, ghi nhận là giao dịch lỗi (Mismatch). Đánh dấu giao dịch đối tác này đã được kiểm tra.

Kịch bản B (Không tìm thấy): Ghi nhận lỗi thiếu giao dịch phía Đối tác (Missing in Partner).

Quét giao dịch mồ côi phía Đối tác (Orphan Partner Transactions):

Java
package com.securebank.service;

import com.securebank.dto.*;
import com.securebank.model.CoreTransaction;
import com.securebank.model.PartnerTransaction;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Service;

import java.time.ZoneId;
import java.time.ZonedDateTime;
import java.util.*;
import java.util.stream.Collectors;

@Slf4j
@Service
public class ReconciliationService {

    private static final ZoneId ZONE_GMT7 = ZoneId.of("Asia/Ho_Chi_Minh");
    private static final ZoneId ZONE_UTC = ZoneId.of("UTC");

    /**
     * Phương thức xử lý đối soát chính giữa Core Banking và Đối tác.
     */
    public List<ReconciliationDiff> reconcile(List<CoreTransaction> coreTxns, List<PartnerTransaction> partnerTxns) {
        log.info("Bắt đầu tiến trình đối soát. Số lượng nhận vào - Core: {}, Partner: {}", 
                coreTxns != null ? coreTxns.size() : 0, partnerTxns != null ? partnerTxns.size() : 0);

        if (coreTxns == null || partnerTxns == null) {
            log.warn("Một trong hai danh sách giao dịch đầu vào bị null. Hủy tiến trình.");
            return Collections.emptyList();
        }

        List<ReconciliationDiff> differences = new ArrayList<>();
        Set<String> matchedPartnerIds = new HashSet<>();

        // Bước 1: Chuẩn hóa và đưa dữ liệu đối tác vào Map để tìm kiếm O(1)
        Map<String, NormalizedTransaction> partnerMap = normalizePartnerTransactions(partnerTxns);

        // Bước 2: Đối soát dựa trên luồng dữ liệu Core Banking
        for (CoreTransaction coreTxn : coreTxns) {
            if (coreTxn == null || coreTxn.transactionId() == null) continue;

            NormalizedTransaction coreNormalized = normalizeCoreTransaction(coreTxn);
            NormalizedTransaction partnerNormalized = partnerMap.get(coreNormalized.transactionId());

            if (partnerNormalized != null) {
                // Kịch bản A: Tìm thấy giao dịch ở cả 2 bên -> Tiến hành so khớp dữ liệu
                matchedPartnerIds.add(partnerNormalized.transactionId());
                verifyMatchingFields(coreNormalized, partnerNormalized, differences);
            } else {
                // Kịch bản B: Giao dịch có ở Core nhưng không thấy ở file Đối tác gửi sang
                recordError(coreNormalized.transactionId(), ReconciliationErrorType.MISSING_IN_PARTNER, 
                        "Giao dịch tồn tại ở Core Banking nhưng thiếu ở hệ thống Đối tác.", differences);
            }
        }

        // Bước 3: Tìm các giao dịch mồ côi (Chỉ có ở đối tác mà Core không ghi nhận)
        findMissingInCore(partnerMap, matchedPartnerIds, differences);

        log.info("Tiến trình đối soát hoàn tất. Tổng số lỗi lệch tìm thấy: {}", differences.size());
        return differences;
    }

    /**
     * Chuẩn hóa dữ liệu Core Banking (GMT+7) về UTC.
     */
    private NormalizedTransaction normalizeCoreTransaction(CoreTransaction coreTxn) {
        ZonedDateTime utcDateTime = coreTxn.transactionDate().atZone(ZONE_GMT7).withZoneSameInstant(ZONE_UTC);
        return new NormalizedTransaction(coreTxn.transactionId(), coreTxn.amount(), coreTxn.status(), utcDateTime);
    }

    /**
     * Chuẩn hóa dữ liệu hệ thống Thẻ Đối tác (GMT+0) về UTC và nhóm lại dạng Map.
     */
    private Map<String, NormalizedTransaction> normalizePartnerTransactions(List<PartnerTransaction> partnerTxns) {
        return partnerTxns.stream()
                .filter(Objects::nonNull)
                .filter(p -> p.txnId() != null)
                .map(p -> {
                    ZonedDateTime utcDateTime = p.txnDate().atZone(ZONE_UTC); // Bản chất GMT+0 tương đương UTC
                    return new NormalizedTransaction(p.txnId(), p.amount(), p.status(), utcDateTime);
                })
                .collect(Collectors.toMap(NormalizedTransaction::transactionId, txn -> txn, (existing, replacement) -> existing));
    }

    /**
     * So sánh chi tiết từng trường dữ liệu nghiệp vụ giữa hai bên.
     */
    private void verifyMatchingFields(NormalizedTransaction core, NormalizedTransaction partner, List<ReconciliationDiff> diffs) {
        // 1. Kiểm tra số tiền giao dịch
        if (Double.compare(core.amount(), partner.amount()) != 0) {
            recordError(core.transactionId(), ReconciliationErrorType.AMOUNT_MISMATCH, 
                    String.format("Sai lệch số tiền. Core: %s, Partner: %s", core.amount(), partner.amount()), diffs);
        }

        // 2. Kiểm tra trạng thái giao dịch
        if (!Objects.equals(core.status(), partner.status())) {
            recordError(core.transactionId(), ReconciliationErrorType.STATUS_MISMATCH, 
                    String.format("Sai lệch trạng thái. Core: %s, Partner: %s", core.status(), partner.status()), diffs);
        }
    }

    /**
     * Phát hiện các giao dịch tồn tại ở Đối tác nhưng Core không có dữ liệu.
     */
    private void findMissingInCore(Map<String, NormalizedTransaction> partnerMap, Set<String> matchedPartnerIds, List<ReconciliationDiff> diffs) {
        for (String partnerId : partnerMap.keySet()) {
            if (!matchedPartnerIds.contains(partnerId)) {
                recordError(partnerId, ReconciliationErrorType.MISSING_IN_CORE, 
                        "Giao dịch tồn tại ở Đối tác nhưng không tìm thấy trong Core Banking.", diffs);
            }
        }
    }

    /**
     * Ghi log tập trung lỗi và đưa vào danh sách chênh lệch kết quả.
     */
    private void recordError(String txnId, ReconciliationErrorType type, String details, List<ReconciliationDiff> diffs) {
        log.error(" [ĐỐI SOÁT LỖI] TxnID: {} | Loại lỗi: {} | Chi tiết: {}", txnId, type, details);
        diffs.add(new ReconciliationDiff(txnId, type, details));
    }
}
3. Giải Thích Thiết Kế Ngắn Gọn
Hiệu năng xuất sắc (O(N+M)): Việc đưa toàn bộ partnerTxns vào HashMap giúp chúng ta loại bỏ hoàn toàn việc phải dùng 2 vòng lặp lồng nhau. Khi duyệt qua danh sách Core, thao tác kiểm tra xem giao dịch có ở hệ thống Visa hay không chỉ tốn thời gian cố định O(1).

Tuân thủ Single Responsibility (SRP): Hàm chính reconcile đóng vai trò điều hướng tổng quan luồng đi (Orchestrator). Các nhiệm vụ cụ thể được bóc tách ra các private method chuyên biệt (normalizeCoreTransaction, verifyMatchingFields, findMissingInCore). Nhờ đó code rất gọn, mạch lạc và dễ dàng viết thêm Unit Test cho từng chức năng độc lập.

Chuẩn hóa thời gian an toàn: Code phân định rõ nguồn múi giờ của từng thực thể (ZONE_GMT7 cho Core và ZONE_UTC cho Partner) trước khi ép chúng về một hệ quy chiếu chung. Điều này triệt tiêu hoàn toàn rủi ro lệch ngày đối soát ở các mốc thời gian biên nhạy cảm (gần nửa đêm).
